# Printfee — дизайн-проект

Сервис печати одежды с собственным дизайном (DTF-печать, Казань, доставка по РФ).
Пакет собран из design-хэндоффа в единый структурированный проект для просмотра и передачи разработчику.

## Публикация на GitHub Pages

Настроено, отдельных действий не требует: пуш в `main` запускает
`.github/workflows/pages.yml`, тот собирает `app/` и публикует.
Один раз нужно включить Pages: **Settings → Pages → Source: GitHub Actions**.

Три вещи, без которых проектный сайт в подпапке `/<имя-репозитория>/` не работает:

- **Базовый путь.** `vite.config.ts` читает `base` из `VITE_BASE`, workflow
  подставляет туда имя репозитория. Локально переменной нет, `base` остаётся
  `/`, и `npm run dev` работает как раньше.
- **Ссылки на файлы из `public/`.** Vite подставляет базовый путь только в то,
  что проходит через сборщик; строковые литералы вида `/assets/logo.svg` в JSX
  и в загрузчиках Three.js он не трогает — они ушли бы в корень домена. Для них
  есть хелпер `src/lib/asset.ts`, **новые ссылки на `public/` писать через него**.
  Роутеру basename передаётся тем же `import.meta.env.BASE_URL`.
- **Прямые заходы на маршруты.** Pages — статика без сервера, и `/catalog`
  вернул бы 404. Workflow кладёт рядом копию `index.html` под именем `404.html`:
  Pages отдаёт её вместо ошибки, дальше адрес разбирает роутер в браузере.

Проверено локальной раздачей сборки из такой же подпапки: главная, прямой заход
на `/catalog` и конструктор с загрузкой 3D-модели.

## Как открыть

Это статические HTML-файлы, для их работы (кастомные теги вроде `<x-dc>`, `{{ }}`-биндинги,
интерактивная логика конструктора) нужен `support.js`, подключённый в каждом файле — поэтому
открывать нужно через локальный сервер, а не двойным кликом по файлу:

```bash
cd printfee-project
python3 -m http.server 8080
```

Затем открыть `http://localhost:8080/` — это `index.html`, точка входа со ссылками на все экраны.

## Структура

```
printfee-project/
├── index.html            — точка входа, ссылки на все экраны
├── mobile.html            — мобильная главная (iOS-фрейм 430×880)
├── desktop.html           — десктопная главная
├── constructor.html       — конструктор принта (рабочий интерактивный прототип)
├── support.js             — рантайм для просмотра прототипов в браузере (НЕ переносить в прод)
├── ios-frame.jsx          — декоративный фрейм устройства для мобильного экрана (не часть дизайна)
├── assets/                — логотипы, мокапы одежды, тех.заметки по обработке фото
│                            (сюда же кладётся assets/tshirt.glb — см. ниже)
├── vendor/three/          — Three.js + GLTFLoader + DecalGeometry (для 3D-сцены конструктора)
└── drafts/                — черновики-эксплорейшн, показанные заказчику в процессе работы
    ├── logo-variants.html
    ├── hero-variants.html
    ├── stats-variants.html
    └── news-animation-variants.html
```

Все три основных экрана и черновики связаны реальными ссылками (десктоп ⇄ конструктор,
`index.html` ⇄ всё). Единственное сознательное исключение — кнопка «Собери свою футболку
прямо в телефоне» на мобильной главной: в макете она помечена бейджем «скоро» и ведёт на
секцию-тизер той же страницы, а не на реальный конструктор — это осознанное состояние
дизайна (мобильный вход в конструктор ещё не спроектирован отдельно), а не забытая ссылка.

## Важно: это не production-код

Файлы — HTML-прототипы для референса. Задача разработчика — воссоздать экраны в целевом
окружении проекта (React/Vue/native — по существующему стеку), используя его паттерны и
стейт-менеджмент, **не копируя HTML/inline-стили напрямую**. `support.js` и `ios-frame.jsx`
нужны только для просмотра прототипа в браузере и в прод не переносятся.

## Дизайн-токены

Десктоп и конструктор — тёмная тема, мобильная главная — светлая (см. ниже).

- Фон (тёмная тема): `#0B0B0C` (десктоп/конструктор)
- Текст: `#EDEDEE`/`#F5F5F4` основной, `rgba(237,237,238,.5-.72)` приглушённый
- Акцент (единственный цветной): коралл `#FF5C38`, hover `#FF7A54`/`#FF8259`
- Шрифты: Onest (десктоп/конструктор), Roboto Mono для лейблов,
  Oswald 700 (`--pf-font-display`) — только мобильная главная
- Радиусы: 10-14px мелкие элементы, 18-28px карточки, 99px пилюли

### Мобильная главная (`app/src/pages/HomePage.css`, `@media (max-width: 780px)`)

Тёмная тема, но своя типографика по референсу [kirarock.team](https://kirarock.team/):
Oswald капсом для заголовков, Roboto Mono для текста и лейблов, срезанные углы
(`clip-path`) вместо скруглений, блок статистики — три строки с волосяными
линиями, новости — свайп-карусель (`scroll-snap`).
Герой (**«Твой принт — твои правила»**) не тронут. Десктоп выше 780px не изменился.

Заголовок и кнопка на мобильном — в том же языке, что и остальной экран:
Oswald капсом в две строки (вторая — коралловым контуром), кнопка не пилюлей,
а срезанным блоком с моноширинным лейблом. Обводка второй строки теперь на
классе `hero__title-stroke`: класс назывался `outline`, а это имя утилиты
Tailwind (`outline-style: solid`) — вокруг строки рисовалась настоящая рамка,
на десктопе тоже.

Волна (`.hero__wave`) лежит не в герое, а в `.hero-screen`: линии идут от
заголовка, проходят за футболкой и гаснут маской ровно там, где начинается
текстовый блок (`.hero-brief`, ~51% высоты `.hero-screen`) — под описанием и
цифрами линий уже нет, иначе текст об них рябит. Стыка между текстом и
футболкой при этом не возникает. На десктопе коробка `.hero-screen` совпадает с героем
(`.pf-morph` там скрыт), так что вид не изменился.
За футболкой поверх волны — коралловое пятно света (`.pf-morph::before`),
кромки растворены маской, чтобы белая майка стояла в свету, а не на линиях.

Края волны в герое (`.hero__wave`) растворяются CSS-маской. Раньше низ волны
замазывался прямо в канве заливкой цвета фона — на десктопе это сходило с рук,
а на мобильном, где канва всего лишь полоса внутри героя, читалось тёмной
ступенью поперёк волны; заливку из `useWaveCanvas` убрал.

Первый экран мобильной главной устроен как плотный бриф: заголовок → футболка
с анимацией → короткое описание → три цифры → кнопка во всю ширину
(`.hero-brief`). Ритм подсмотрен у [4-6.digital](https://4-6.digital/) —
композиционная схема, не оформление: визуал, шрифты, коралл, срезанные углы и
живая волна свои. Раньше первый экран отдавал только слоган и картинку, а всё
доказательное уезжало вниз.

Блок сознательно чуть длиннее окна: на 390×844 кнопка попадает в экран впритык
(834 из 844), на низких экранах уезжает за фолд — это нормально, край блока
зовёт листать. Футболка ужимается по трём осям сразу — `min(62vw, 236px, 26svh)`,
последнее слагаемое и держит кнопку в пределах экрана на высоких телефонах.
У `.pf-morph` сверху 40px паддинга: мем стартует выше ворота (`yPercent -118`
в `useMorphLoop`) и без запаса налезал на заголовок.

Цифры и описание на мобильном живут в `.hero-brief`, на десктопе — прежними
секциями `.hero-intro` и `.stats-row`, скрытыми на мобильном через
`.pf-mob-hide`. Значения цифр берутся из одного массива `stats` в `HomePage.tsx`,
рендерятся дважды — дублируется разметка, но не данные.

Под кнопкой «Создать принт» — петля «фото → футболка» (`.pf-morph`): мемы
`morph-print-1..3.jpg` по очереди влетают фотокарточкой и ложатся принтом на
`morph-tee.png` (цикл 21с = 3 × 7с). Вместо гифки (гиф той же длины — 5-8 МБ,
тут только сами картинки); белый фон мема гасится об ткань через
`mix-blend-mode: multiply`, который переключается с `normal` на кадре касания —
в полёте карточка висит над воротом на тёмном фоне, где белая бумага и должна
читаться. Если снимут реальное видео процесса — блок меняется на `<video>`.

Расписание петли — **единственное место в проекте на GSAP** (`useMorphLoop`):
нужны упругая отдача на приземлении (принт сплющивается и отпружинивает,
футболка отыгрывает удар), своя траектория у каждого мема и остановка петли,
когда блок ушёл за экран. На `@keyframes` это было расписание из магических
процентов, которое нельзя ни притормозить, ни переиграть. Остальные анимации
главной остались на CSS сознательно — там нет ничего, чего CSS не умеет.
Тонкость GSAP: у всех `fromTo` стоит `immediateRender: false`, иначе начальное
состояние применяется ещё при сборке таймлайна и футболка висит сплющенной
до первого проигрывания.

Внизу мобильной главной — липкая панель заказа (`.pf-stickycta`), появляется,
когда герой с кнопкой уходит за экран. **Цена в ней — заглушка**: константа
`PRICE_FROM` в `HomePage.tsx`, настоящую нужно взять у заказчика. Отступ подвала
под панель поднимается через `body:has(.pf-stickycta--in)`, а не безусловным
правилом — подвал общий для всех страниц, а панель есть только на главной.

Появление блоков при скролле — `.reveal` / `.reveal-stagger`, на нативном
`animation-timeline: view()` (без JS). Прежний вариант на `IntersectionObserver`
удалён вместе с хуком `useInView`: при переходе по якорю `/#news` вьюпорт
перепрыгивает блок, событие пересечения не приходит, и блок навсегда оставался
невидимым. В браузерах без scroll-timeline (Firefox) блоки просто видны сразу.

**Анимация двигает, но не прячет.** Изначально в `pf-reveal` была ещё и
`opacity: 0 → 1`, то есть видимость контента решала анимация. При высоте окна
больше содержимого (планшет в альбомной, превью ссылки в мессенджере, печать в
PDF, headless-рендер) секции оказывались внутри вьюпорта на нулевой прокрутке,
их диапазон не начинался — и половина страницы рендерилась пустой. Теперь
анимируется только `translateY`: худший случай — контент виден без сдвига.

## Известный technical debt

- Худи и свитшот временно скрыты в выборе товара конструктора (только футболка активна) —
  вся логика для них в коде осталась, но нужны качественные PNG-мокапы перед/спина от заказчика.
- Мокап футболки/свитшота — временная векторная отрисовка (`assets/mock-*.svg`) вместо фото,
  стоит заменить на фото-мокапы при получении нормального сета от заказчика.

## 3D-модель футболки в конструкторе

Сцена конструктора отрисовывается через Three.js (`vendor/three/`, подключён напрямую как
классические `<script>`, без сборки) вместо старого трюка «плоский мокап + CSS-сквош».
Вращение мышью (yaw + лёгкий pitch), перекраска и перенос принта на поверхность модели
(через `THREE.DecalGeometry`, проекцией из 2D-координат слоя на ближайшую точку меша) —
реализованы, подключены и проверены сквозным тестом (загрузка, вращение, перекраска,
добавление текстового принта — всё воспроизведено в реальном браузере на самой модели).

**Модель подключена:** `assets/tshirt.glb` (~19.7 МБ, ~25k вершин/~41k треугольников).
Собрана из присланных исходников `assets/3d/fbx/fbx_0.fbx` + `fbx_diffuse_1001.png` +
`fbx_normal_1001.png` (FBX → GLB через Three.js `FBXLoader`/`GLTFExporter` в браузере) —
исходная конвертация дала ~577k вершин/~1.02M треугольников (~111 МБ), из-за чего
пересборка одной декали принта занимала **1.3–2.7 секунды** (рейкастинг и построение
геометрии декали на такой плотной сетке) — конструктор ощутимо лагал при любом
перетаскивании/добавлении принта. Упростил геометрию через `@gltf-transform` +
`meshoptimizer` (тот же алгоритм, что использует экспорт из Blender/Unity, с сохранением
UV/нормалей) — после упрощения та же операция занимает **20–85 мс**, визуально модель не
отличить (сохранены складки ткани и общий силуэт). Оригинальный тяжёлый файл сохранён как
`assets/tshirt.original.glb` — на случай, если понадобится более высокое качество сетки
для чего-то другого (например, макросъёмки/рекламных рендеров вне конструктора).

Второй присланный файл, `assets/3d/fbx/fbx_1.fbx`, не использовался (в исходниках было
два варианта без явных имён/пометок — если это другая поза/раскрой, а не дубль, стоит
уточнить у заказчика и пересобрать при необходимости).

**Известные ограничения текущей сборки:**
- Текстуры материала в исходниках — 4096×4096 (`fbx_diffuse_1001.png`/`fbx_normal_1001.png`).
  Диффуз-карта **сознательно отключена** в `onModelLoaded` (`m.map = null`) — это не цвет
  ткани, а техническая раскладка лекал (выкройка серым по белому фону), при натяжении на
  модель она выглядела как рваные заплатки; используется только normal-карта (лёгкая
  фактура), а цвет полностью управляется плашкой «Цвет». Обе текстуры можно смело ужать
  до 2048 без потери качества на экране, если нужно урезать вес ещё сильнее.
- Раскраска (плашка «Цвет» в шапке сцены) тонирует **все** материалы модели через
  `material.color` — в этом файле материал один, но если пришлют версию с раздельными
  материалами (ткань/шов/бирка), перекраску нужно будет сузить по имени материала в
  `onModelLoaded` (`constructor.html`).
- Лицензия/источник исходных FBX не зафиксированы в этом README — если модель делали не
  сами (аутсорс/сток), стоит дописать сюда автора и права.

Позиция и размер декали (`Component.buildDecalMesh`) считаются из реальной геометрии
камеры и блока сцены (FOV, дистанция камеры, размер `stage`-контейнера в px) — те же
координаты x/y (%) и `scale`, что и у плоского оверлея редактирования, поэтому то, что
двигаешь/растягиваешь при редактировании, всегда совпадает с тем, что реально ложится на
модель. Ручной калибровки под конкретный файл почти не требуется, кроме:
- `FRONT_FACING_Z` (сейчас `1`, `constructor.html`) — если принт для «переда» окажется на
  спине модели, поменять на `-1`.
- `_decalBaseSize` (`Component.onModelLoaded`) — влияет только на глубину проекции декали
  (насколько она «оборачивается» по кривизне ткани), не на видимый размер принта.

## Источник

Собрано из `design_handoff_printfee/` и черновиков `Printfee_ дизайн сайта/` — те исходные
папки не изменялись, этот проект — их структурированная, связанная навигацией копия.
